## 1. 接口原则

- 前台不直连 Polymarket / Predict.fun，统一调用 GemW 聚合后端。
- 所有交易接口必须绑定具体 `TradeableInstrument`，每个标的只绑定一个底层平台、链、资产和订单接口。
- `EventGroup` 只用于同主题展示、归组推荐、持仓归组和套利提示，不作为交易对象，不合并盘口、不合并深度、不合并持仓成本池。
- 原平台市场进入本平台前，必须先经过“标的过滤规则”的评分、强制过滤和人工审核分流。
- 跨平台事件归组前，必须先经过“事件归组规则”的匹配度评分；只有达到高匹配阈值才允许归组，否则不归组。
- Quote 必须带有效期、价格版本、费用预估、滑点和预计到手收益。
- 下单前必须校验钱包、链、余额、授权、签名、行情、平台健康、地区限制和标的状态。
- 每笔聚合订单只映射一个底层平台订单，不做跨平台拆单、不做智能路由、不做托管余额。
- 第三方 API 请求失败可自动重试，仍失败后降级或熔断；降级、暂停、恢复必须写入审计日志。
- 规则配置、规则试算、人工审核、暂停交易、推荐上下架、结算复核等关键后台动作必须记录操作人、原因、影响范围和规则版本。

## 3. 市场同步与 API 管理

对应后台模块：`市场同步`。

### 3.1 获取 API 健康列表

`GET /api/admin/market-sync/api-health`

用途：支撑“API 管理与降级操作”列表。

入参：

| 字段 | 说明 |
| --- | --- |
| `venue` | Polymarket / Predict.fun，可选 |
| `status` | healthy、stable、degraded、paused、rate_limited、auth_failed |
| `keyword` | 搜索平台、API、影响范围 |

出参：

- `apiHealthId`
- `venue`
- `apiName`，例如 Gamma / Data / CLOB、Orders、WebSocket
- `status`
- `latency`
- `successRate`
- `lastError`
- `affectedInstrumentCount`
- `affectedOrderCount`
- `suggestion`
- `availableActions`

规则：

- 总览页只展示轻量健康摘要；API 详情、重试、暂停、日志和心跳统一在市场同步页处理。
- `availableActions` 由当前状态决定。例如 healthy 展示 `view`、`detect`、`logs`；degraded 展示 `view`、`retry`、`pause_ordering`、`failed_orders`。

### 3.2 查看 API 健康详情

`GET /api/admin/market-sync/api-health/{apiHealthId}`

用途：点击“查看”或“心跳”时打开详情弹窗。

出参：

- API 基础信息：平台、API 名称、用途、负责服务。
- 健康指标：成功率、延迟、频控次数、最近错误、最近检测时间。
- 影响范围：标的、智能推荐、订单、持仓、前台入口。
- 可执行动作：立即检测、手动重试、暂停下单、手动重连、降级轮询、查看日志。
- 最近日志。

规则：

- 查看详情不改变业务状态。
- 详情弹窗不展示“待处理”状态标签，只展示当前状态、影响范围和可执行动作。

### 3.3 立即检测 API

`POST /api/admin/market-sync/api-health/{apiHealthId}/detect`

用途：点击“立即检测”后直接提交检测任务。

入参：

- `reason`

出参：

- `taskId`
- `taskStatus`
- `message`

前端反馈：

- `Polymarket Gamma / Data / CLOB 正在检测，完成后会刷新状态和影响范围。`

规则：

- 检测认证、延迟、频控、订单提交能力和行情新鲜度。
- 不暂停交易，不修改标的状态。
- 操作写入审计日志。

### 3.4 手动重试 API

`POST /api/admin/market-sync/api-health/{apiHealthId}/retry`

用途：对 degraded、rate_limited、api_error 的接口提交一次人工重试。

出参：

- `taskId`
- `retryScope`
- `affectedInstrumentCount`
- `message`

规则：

- 重试对象是当前 API 资源，不是全部平台。
- 重试完成后刷新状态、影响范围和最近错误。

## 4. 标的过滤规则

对应后台模块：`标的过滤规则`。

### 4.1 获取标的过滤规则配置

`GET /api/admin/instrument-filter/rules`

出参：

- `ruleVersion`
- `scoreWeights`
- `scoringRules`
- `thresholds`
- `forceRules`
- `lastUpdatedBy`
- `lastUpdatedAt`

`scoreWeights` 至少包含：

- 流动性
- 成交量 / 热度
- 规则清晰度
- 到期时间
- API 可交易性

不包含：

- 合规 / 敏感风险。该类风险由强制过滤规则和风控审计处理，不参与推荐分加权。

权重规则：

- 每个参数权重范围默认 `0-60`。
- 全部权重合计必须等于 `100`。
- 前端滑杆变化后，应实时回传草稿值并刷新公式展示；后端保存时仍需再次校验。

`scoringRules` 至少包含 5 个评分项，每项包含：

- `factor`：评分参数，例如 liquidity、heat、rule_clarity、expiry_time、api_tradability。
- `factorName`：后台展示名称。
- `weight`：权重。
- `sourceFields`：参与计算的标准化接口字段。
- `scoreRange`：单项分范围，固定为 0-100。
- `scoreBands`：评分区间配置，例如字段条件、分值范围、说明。
- `ruleType`：分段阈值、时间窗口、API 状态、组合条件等。
- `ruleDescription`：该参数如何计算单项分。
- `hardCap`：特殊情况下的最高分限制，例如盘口 stale 时流动性分不高于 40。
- `sample`：用于后台展示的样例说明。

评分项字段来源示例：

| 评分项 | 字段来源 |
| --- | --- |
| 流动性 | `liquidityUsd`、`depthUsdWithin2Pct`、`bestBid`、`bestAsk`、`spreadBps`、`orderbookUpdatedAt` |
| 成交量 / 热度 | `volume24hUsd`、`tradeCount24h`、`openInterestUsd`、`venueHotRank`、`watchlistCount` |
| 规则清晰度 | `hasResolutionSource`、`hasResolutionRule`、`hasEndTime`、`hasOutcomeMapping`、`ruleParseStatus`、`descriptionCompleteness` |
| 到期时间 | `endTime`、`hoursToClose`、`settlementWindowHours`、`isTradingClosed` |
| API 可交易性 | `orderbookStatus`、`quoteAvailable`、`canPlaceOrder`、`canCancelOrder`、`canQueryOrder`、`canQueryPosition`、`authStatus`、`chainSupported`、`collateralSupported` |

`thresholds` 至少包含：

- `autoAdmitScore`：自动进入可交易标的阈值，例如 80。
- `manualReviewMinScore`：人工审核下限，例如 60。
- `rejectBelowScore`：低于该分数直接过滤。
- `smartRecommendMinScore`：进入智能推荐候选池阈值，例如 85。

`forceRules` 至少包含：

- `ruleId`
- `ruleName`
- `action`：filter、manual_review。
- `triggerMode`：any_match，表示任一启用条件命中即触发；高级组合规则可二期扩展。
- `enabledConditions`
- `keywordFilter`：是否启用敏感关键词过滤、关键词数组、命中字段。
- `categoryFilter`：是否启用分类过滤、分类数组。
- `expiryFilter`：是否启用到期时间过滤、早于时间、晚于时间。
- `unclearRuleFilter`：是否启用规则不清晰过滤。启用后固定识别结算来源缺失、描述无法解析、结果定义歧义。
- `status`

`keywordFilter.matchFields` 固定支持：

- `rawTitle`
- `normalizedTitle`
- `description`
- `tags`
- `category`
- `resolutionRuleText`

不支持：

- `regionFilter`。原因是底层平台标的通常不提供稳定的结构化地区字段；地区类风险如出现在文本中，应通过敏感关键词过滤处理。

### 4.2 保存标的过滤规则配置

`PUT /api/admin/instrument-filter/rules`

入参：

- `scoreWeights`
- `scoringRules`
- `thresholds`
- `forceRules`
- `reason`

规则：

- 强制过滤规则优先级高于推荐分。
- 每个评分参数都必须包含单项评分规则和权重，不能只保存权重。
- 全部权重合计必须等于 100%，否则返回 `RULE_CONFIG_INVALID`。
- 单项权重超过后台配置上限时，返回 `RULE_CONFIG_INVALID`。
- 强制过滤规则不承载流动性和 API 异常判断；流动性归属评分模型，API 异常归属市场同步、API 认证和风控模块。
- 新增强制过滤规则至少启用一个条件，否则返回 `RULE_CONFIG_INVALID`。
- 命中“直接过滤”时，即使推荐分较高，也不能进入可交易标的。
- 命中“进入人工审核”时，不能自动准入。
- 保存后生成新 `ruleVersion`，并返回浏览态可展示的生效版本。
- 保存必须写入审计日志，记录操作人、变更字段、旧值、新值、试算影响范围和生效时间。

### 4.2.1 保存单个评分项配置

`PUT /api/admin/instrument-filter/scoring-rules/{factor}`

用途：点击某个评分项的“编辑”后，只保存该评分项草稿配置。

入参：

- `factor`
- `sourceFields`
- `weight`
- `scoreBands`
- `ruleDescription`
- `hardCap`
- `sample`
- `reason`

规则：

- 保存单个评分项后进入规则草稿，不立即影响线上同步。
- 最终仍需通过“保存规则”生成新 `ruleVersion` 后才正式生效。
- 后端需要校验 `sourceFields` 必须来自标准化市场字段白名单。
- `scoreBands` 的分值区间必须在 `0-100` 内。

### 4.2.2 强制过滤规则 CRUD

新增强制过滤规则：

`POST /api/admin/instrument-filter/force-rules`

查看强制过滤规则：

`GET /api/admin/instrument-filter/force-rules/{ruleId}`

编辑强制过滤规则：

`PUT /api/admin/instrument-filter/force-rules/{ruleId}`

删除强制过滤规则：

`DELETE /api/admin/instrument-filter/force-rules/{ruleId}`

规则：

- 新增、编辑、删除都只进入规则草稿，不立即影响线上同步。
- 删除需要写入审计日志，记录删除人、规则 ID、规则名称和影响试算结果。
- 强制规则至少启用一个条件。
- 强制规则只承载敏感关键词、分类、到期时间、规则不清晰 4 类条件。

### 4.5 审核标的准入

`POST /api/admin/instrument-filter/review-queue/{reviewId}/review`

入参：

| 字段 | 说明 |
| --- | --- |
| `action` | approve_as_tradeable、reject、force_filter |

- 审核详情必须能解释推荐分、强制规则命中、字段来源和通过/拒绝后的业务影响。
- 操作写入审计日志，并保留规则版本。

## 6. 事件归组规则

对应后台模块：`事件归组规则`。

### 6.1 获取事件归组规则配置

`GET /api/admin/event-group-rules`

出参：

- `ruleVersion`
- `matchWeights`
- `scoringRules`
- `thresholds`
- `lastUpdatedBy`
- `lastUpdatedAt`

`matchWeights` 至少包含：

- 主题语义相似
- 结果定义一致
- 结算时间一致
- 结算规则一致
- 交易参数一致性

不包含：

- 平台资产差异。链、资产、底层平台是展示和交易提示字段，不参与“是否同一事件”的匹配度评分。

`scoringRules` 至少包含：

- `factor`：semantic_similarity、outcome_definition、settlement_time、settlement_rule、trade_parameter。
- `factorName`
- `weight`
- `sourceFields`
- `scoreBands`
- `ruleDescription`
- `hardCap`
- `sample`

归组评分项字段来源示例：

| 匹配项 | 字段来源 |
| --- | --- |
| 主题语义相似 | `normalizedTitle`、`rawTitle`、`description`、`category`、`tags`、`semanticEmbeddingScore` |
| 结果定义一致 | `outcomeName`、`outcomeSide`、`outcomeDefinition`、`thresholdValue`、`comparisonOperator`、`unit` |
| 结算时间一致 | `endTime`、`closeTime`、`resolutionTime`、`timezone`、`settlementWindowHours` |
| 结算规则一致 | `resolutionSource`、`resolutionRuleText`、`oracleSource`、`dataProvider`、`tieBreakRule`、`ruleParseStatus` |
| 交易参数一致性 | `probability`、`bestAsk`、`bestBid`、`tradableStatus`、`marketType`、`collateralToken`、`chainId` |

`thresholds` 至少包含：

- `groupMinScore`：允许归组阈值，例如 95。
- `notGroupBelowScore`：低于该分数不归组。

### 6.2 保存事件归组规则配置

`PUT /api/admin/event-group-rules`

入参：

- `matchWeights`
- `scoringRules`
- `thresholds`
- `reason`

规则：

- 每个匹配项都必须包含字段来源、分值区间和权重，不能只保存权重。
- 权重合计必须等于 100%，否则返回 `RULE_CONFIG_INVALID`。
- 事件归组不设置强制规则；结果定义、结算规则、到期时间等差异通过匹配项自然降分。
- 只有匹配度达到 `groupMinScore` 才允许归组，低于阈值直接不归组。
- 一期任何 EventGroup 都不得合并盘口深度。
- 保存后生成新 `ruleVersion`，写入审计日志。

### 6.2.1 保存单个归组匹配项配置

`PUT /api/admin/event-group-rules/scoring-rules/{factor}`

用途：点击某个匹配项的“编辑”后，只保存该匹配项草稿配置。

入参：

- `factor`
- `sourceFields`
- `weight`
- `scoreBands`
- `ruleDescription`
- `hardCap`
- `sample`
- `reason`

规则：

- 保存单个匹配项后进入规则草稿，不立即影响线上归组任务。
- 最终仍需通过“保存规则”生成新 `ruleVersion` 后才正式生效。
- 后端需要校验 `sourceFields` 必须来自标准化事件/标的字段白名单。
- `scoreBands` 的分值区间必须在 `0-100` 内。

| --- | --- |
| `action` | publish、pause、resume、delist、update_copy、update_rank、reject |
| `copy` | 修改推荐文案时传 |
| `riskCopy` | 修改风险文案时传 |
| `rank` | 调整排序时传 |
| `reason` | 操作原因 |

规则：

- `pause` 是临时暂停展示，可恢复。
- `delist` 是下架当前推荐机会，通常需要重新进入审核后才能上线。
- `update_copy` 只修改文案，不改变标的交易状态。
- `update_rank` 只调整展示优先级。
- 所有动作写入审计日志。

入参：
