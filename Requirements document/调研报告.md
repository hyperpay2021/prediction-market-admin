# 预测市场聚合器调研报告

版本：v0.2

更新时间：2026-06-03

调研对象：

- Polymarket：https://polymarket.com/
- Predict.fun：https://predict.fun/zh-cn/markets
- agg.market：https://agg.market/events
- 本地需求资料：`预测市场产品规划.pdf`、`Prediction market聚合器技术方案.pdf`

## 1. 结论摘要

一期产品应收敛为：

> Polymarket + Predict.fun 的标的聚合交易入口，先解决“发现优质标的、弱化平台选择、用 Web3 钱包直接交易、让小白快速下单”的问题。

这意味着一期不是跨平台订单簿聚合器，也不是券商式托管交易平台。一期只做两个平台事件/标的的采集、筛选、归组、展示和单平台绑定交易：

- 同一预测主题在 Polymarket 和 Predict.fun 都存在时，可以归组或相似推荐，但仍展示为两个独立可交易标的。
- 每个可交易标的绑定唯一底层平台、链、资产和订单接口。
- 用户投资哪个标的，就把订单下到该标的对应平台。
- UI 不突出平台品牌，但订单详情、钱包切链、签名、风控和审计必须保留底层平台信息。
- Maker/Taker 都进入一期，但都基于“当前选中的具体标的”执行，不做跨平台拆单、不自动选择其他平台、不复制挂单。
- 用户用 EVM 钱包连接平台，EVM 地址作为用户唯一 ID；平台不托管资产、不帮用户开上手账户、不做 managed balance。
- 产品体验重点是快捷交易面板和 AI 自动报单，让用户像使用简版期权/二元期权一样快速理解并下单。

二期再考虑盘口深度合并、大额拆单、跨平台最优路由、自有平台撮合、做市商和自动资金调度。

## 2. 本地需求文档校准

### 2.1 产品规划 PDF

产品规划图中一期包含：

- 聚合各平台报价。
- 单平台交易。
- Maker 到各平台。
- Taker 吃各个平台的单。
- 自动报单，优先考虑流动性好的一侧。
- UI 支持 PC 专业用户与移动端小白用户切换。
- 支持 Polygon、BSC 链。

结合最新产品决策，这里的“一期聚合”应理解为标的和报价入口聚合，不是盘口深度合并：

- Polymarket 事件 A 与 Predict.fun 事件 B 如果是同一预测主题，一期仍作为两个可交易标的存在。
- 平台露出被弱化，但标的和底层平台是一一绑定的。
- Maker 挂单、Taker 吃单都发生在该标的绑定的平台。
- 自动报单是一期关键能力，用于把热门、高流动性、赔率容易理解的上手平台标的包装成快捷交易入口。

### 2.2 技术方案 PDF

技术方案中提到的数据接入、事件对齐、行情同步、交易执行、链支持和风控仍然适用于一期，但以下能力应降级为二期：

- 聚合订单簿。
- 多平台盘口深度合并。
- 跨平台拆单。
- 智能订单路由 SOR。
- 跨链自动换币/rebalance。
- 代付 Gas。
- 自有撮合和做市商。

一期技术重点应改为：

- 原始平台市场采集和清洗。
- 事件/标的归组，但不合并盘口。
- 单标的单平台交易适配。
- Web3 钱包连接、授权、签名、自动切链。
- AI 热门标的筛选和自动报单。
- 快捷/专业双交易面板。
- 订单、持仓、审计日志统一展示，但底层记录仍按平台拆开。

## 3. 平台调研

### 3.1 Polymarket

定位：全球主流预测市场，事件覆盖广、流动性强，是一期必须优先接入的平台。

技术特征：

- 官方 API 分为 Gamma API、Data API、CLOB API。
- Gamma API 负责市场、事件、标签、系列、搜索等发现类数据。
- Data API 负责用户持仓、交易、活动、持有人数据、排行榜等分析类数据。
- CLOB API 负责订单簿、价格、midpoint、spread、历史价格以及下单、撤单等交易操作。
- CLOB 是链下订单撮合 + 链上原子结算；订单为 EIP-712 签名消息，在 Polygon 上结算。
- 交易认证涉及 L1 钱包签名和 L2 API 凭证；下单需处理 signature type、funder address、订单提交、撤单和状态同步。
- 官方提供 TypeScript、Python、Rust SDK。

对一期的影响：

- 事件和市场发现优先走 Gamma API。
- 标的详情需要读取该标的自身盘口/价格，但不与 Predict.fun 盘口合并。
- 用户在 Polymarket 标的上下单时，钱包需要切到 Polygon 并使用对应 USDC/授权状态。
- 平台需要通过 API 帮用户生成签名 payload、提交订单、查询订单和持仓，但资产仍在用户 Web3 钱包和底层协议中。
- 需要按官方频控设计后端缓存、重试和熔断，避免前端直接高频打第三方 API。

官方资料：

- API Overview：https://docs.polymarket.com/api-reference/introduction
- Trading Overview：https://docs.polymarket.com/trading/overview
- Rate Limits：https://docs.polymarket.com/api-reference/rate-limits

### 3.2 Predict.fun

定位：BNB Chain 原生预测市场，适合作为 Polymarket 之外的第二流动性来源，也契合 BNB/币安生态用户。

技术特征：

- Predict REST API 支持程序化交易。
- Mainnet API base URL：`https://api.predict.fun/`
- Testnet API base URL：`https://api-testnet.predict.fun/`
- 主网 API 请求需要 API Key。
- 支持 TypeScript SDK `@predictdotfun/sdk` 和 Python SDK。
- 支持传统 EOA，也支持 Smart Wallet / Predict Account。
- 订单创建流程包含 USDT 与 ConditionalTokens 授权、构建订单、生成 typed data、签名订单、计算订单 hash、提交订单。
- 支持 LIMIT order 和 MARKET order；官方推荐 LIMIT order。
- Predict 盘口以 Yes outcome 为基准，No outcome 需要通过互补价格计算，且要注意 decimalPrecision。
- WebSocket endpoint：`wss://ws.predict.fun/ws`，公共市场数据、盘口和价格更新不需要认证；用户私有流需要 JWT。
- 默认主网 API Key 频控为 240 requests/min。

对一期的影响：

- Predict.fun 标的绑定 BSC/BNB Chain 和 USDT。
- 标的价格、盘口和成交信息只用于该标的详情和交易面板，不参与跨平台深度合并。
- No 侧价格展示需要基于 Predict.fun 官方规则做精度转换，避免快捷面板出现错误赔率。
- 用户在 Predict.fun 标的下单时，钱包需要切到 BSC/BNB Chain，检查 USDT 余额、授权和签名能力。
- Predict.fun API 仍有 Beta 风险，需要用独立适配层隔离接口变化。

官方资料：

- Predict REST API：https://docs.predict.fun/developers/predict-rest-api
- Predict API Reference：https://dev.predict.fun/
- Orderbook Guide：https://dev.predict.fun/understanding-the-orderbook-685654m0
- Create / Cancel Orders：https://dev.predict.fun/how-to-create-or-cancel-orders-679306m0
- WebSocket：https://dev.predict.fun/general-information-1915499m0

### 3.3 agg.market

定位：预测市场聚合器竞品，公开文档显示其覆盖 canonical market matching、omni-chain funding、聚合订单簿、跨场所/跨链路由、SDK/UI 组件等能力。

对我们的启发：

- agg.market 的二期/远期能力很完整，但一期不应照搬其跨场所执行和资金系统。
- 它验证了“市场匹配 + 交易组件 + 账户体验”这个方向，但我们的短期落地应更轻：不托管、不合并盘口、不跨平台路由。
- 我们的一期差异点应放在中文/亚洲用户体验、BNB Chain + Polygon 的钱包自动切链、AI 热门标的自动报单、快捷交易面板。
- 后台仍需要保留 venue attribution，因为即使前台弱化平台，每个订单、签名、链、资产和失败原因都与底层平台绑定。

## 5. 一期产品机会

### 5.1 用户痛点

- 用户不知道哪些预测市场标的值得交易。
- 同一主题可能在多个平台出现，但用户不想理解平台差异。
- 新用户不理解 Yes/No、概率价格、最大亏损、结算规则、撤单、未成交等概念。
- Web3 用户不想开多个平台账户，也不想把资金托管给新平台。
- Polymarket 和 Predict.fun 在链、资产、授权、签名、订单状态上差异大。
- 专业用户需要盘口和限价挂单；小白用户更需要“一句话解释收益 + 一键下单”。

### 5.2 产品定位建议

一期定位：

> 面向预测市场用户的 Polymarket + Predict.fun 标的聚合交易入口，用 AI 推荐优质标的，用快捷/专业双面板完成单平台绑定交易。

体验原则：

- 默认弱化平台品牌，但不抹掉底层平台事实。
- 标的是交易最小单元，标的绑定唯一平台、链、资产、市场 ID 和订单接口。
- 快捷模式优先，专业模式补充。
- 不托管资产，不代用户开设底层平台账户。
- 价格、概率、到期时间、最大亏损、结算条件必须让小白看得懂。
- Maker/Taker 都支持，但都只针对当前选中的标的。

## 6. 一期核心风险

### 6.1 事件归组风险

两个平台的事件可能标题相似但结算规则不同。由于一期不合并盘口，错误归组不会直接造成跨平台拆单错误，但仍会误导用户把不同标的理解成同一主题。

建议：

- 使用 AI 做候选归组和相似推荐，但不能把归组当成盘口合并。
- 归组页必须保留两个可交易标的的独立结算规则、到期时间和价格。
- 高流量/高推荐标的需要人工复核归组和文案。

### 6.2 单标的价格展示风险

Predict.fun 的 No 侧价格需要互补计算，Polymarket 的 outcome token 也需要映射清楚。即使一期不做盘口合并，价格方向和单位错误仍会直接影响快捷下单。

建议：

- 内部按单平台 `TradeableInstrument` 标准化 outcome、price、size、collateralToken。
- 每个价格必须保留 venue、marketId、outcomeId、更新时间和精度。
- 快捷文案必须引用最新 quote，并设置有效期。

### 6.3 交易执行风险

下单涉及钱包切链、授权、签名、平台 API、盘口变化。报价到提交之间价格可能变化，导致失败或成交劣化。

建议：

- Quote 必须有有效期。
- Taker 必须有最大滑点保护。
- Maker 必须有明确价格、数量、有效期和撤单能力。
- 每笔订单只映射一个底层平台订单，降低一期执行复杂度。

### 6.4 钱包与网络风险

用户连接的是一个 EVM 钱包，但 Polymarket 和 Predict.fun 分别使用 Polygon 和 BSC/BNB Chain。自动切链失败、用户未添加网络、资产不足、授权不足都会打断下单。

建议：

- 钱包连接层支持 WalletConnect 类 EVM 钱包。
- Polymarket 标的下单前自动切 Polygon。
- Predict.fun 标的下单前自动切 BSC/BNB Chain。
- 切链失败、未安装链、余额不足、授权不足时阻止下单并给出明确提示。

### 6.5 API 稳定性与频控风险

两个平台 API 都有频控或稳定性风险。推荐标的、快捷报价和订单查询都不能依赖前端直连第三方。

建议：

- 后端 Worker 统一接入第三方 API，并缓存市场、价格、盘口摘要和订单状态。
- 每个平台建立独立限流、重试、熔断和健康状态。
- 自动报单标的在 API 异常、盘口 stale、流动性下降或临近结算异常时自动下架。

### 6.6 合规风险

预测市场在不同地区有访问限制。技术方案已提到需要限制敏感地区 IP 访问交易功能。

建议：

- 一期至少做交易功能地理限制、服务条款确认、风险提示。
- 管理后台记录用户 EVM 地址、下单 IP、地区判断、签名 hash、执行平台、订单回执。

## 7. 一期建议范围

### 7.1 MVP 必须包含

- 两平台市场/事件/标的同步。
- 事件主题归组：AI 候选 + 人工复核，不合并盘口。
- 标的列表和详情页：每个标的绑定唯一平台、链、资产和订单接口。
- 快捷交易面板：面向小白，支持一键买 Yes/No。
- 专业交易面板：支持 Market/Limit、价格、数量、盘口、撤单。
- Taker 下单：当前标的单平台提交。
- Maker 挂单：当前标的单平台提交和撤单。
- Web3 钱包登录：EVM 地址作为用户 ID。
- 自动切链：Polygon / BSC。
- 订单、成交、持仓查询：通过底层平台 API 和钱包地址聚合展示，但不做托管余额。
- AI 自动报单：热门、高流动性、规则清晰标的推荐。
- 风控：地区限制、quote 有效期、滑点保护、授权/余额检查、平台 API 熔断、推荐标的下架。

### 7.2 一期增强项

- 首页推荐 banner / 自动报单窗口。
- 快捷面板文案模板和 A/B 测试。
- 专业模式盘口深度、成交历史、订单有效期。
- 订单和持仓按“同主题”归组展示，但不合并成本池。
- 后台 AI 推荐标的管理：上线、下架、文案审核、推荐优先级、风险暂停。
- 成交、部分成交、撤单、结算通知。

### 7.3 不进入一期

- 跨平台盘口深度合并。
- 聚合订单簿。
- 跨平台最优路由。
- 大额拆单。
- 同一 Maker 订单多平台复制挂单。
- 托管账户 / managed balance。
- 自动跨链换币 / rebalance。
- 代付 Gas。
- 自研预测市场。
- 自研 Oracle。
- 无风险套利 MM 产品化。
- 对外开放成交 API。
- 暗池撮合。

## 9. 后续需要确认的问题

- 一期主网交易是否先做白名单灰度。
- 一期是否必须支持卖出/平仓，还是先支持买入、挂单、撤单。
- 钱包连接优先使用 WalletConnect、GemW、Passkey 钱包，还是多方案并行。
- Polymarket 和 Predict.fun 的 API Key、签名凭证、测试环境是否已准备。
- 自动报单的推荐阈值：最低流动性、最低成交量、最短/最长到期时间、最低赔率可读性。
- 地区限制名单和合规文案由谁确认。
